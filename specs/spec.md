# fa - Harness Engineering Tool Specification

## Overview

`fa` is a task orchestration tool that automates running AI coding agents on defined tasks. It manages task lifecycle, dependencies, multi-attempt workflows, and policy-based verification.

## Installation

```bash
# Install as editable uv tool
uv tool install --editable /path/to/fa fa
```

## CLI Structure

```
fa init                                    # Initialize .fa/ structure
fa task create <slug> [--parent <id>]      # Create task (optionally as subtask)
fa task list [--status <status>]           # List tasks
fa task info <id>                          # Show task details
fa task done <id|range>                    # Mark tasks completed
fa task rm <id|range> [-f]                 # Delete tasks
fa task archive <id|range>                 # Archive completed tasks
fa task run [options]                      # Execute pending tasks
fa policy list                             # List available policies
fa policy run <policy-ids...> [options]    # Run one or more policies
fa policy run-all [options]                # Run all policies
```

### Task Run Options

```
fa task run
  [--start <id>]           # Start from task ID
  [--end <id>]             # End at task ID
  [--tool <tool>]          # AI tool: kilo|iflow|claude|ccr|opencode (default: iflow)
  [--rounds <n>]           # Rounds per task (default: 3)
  [--policies <id1,id2>]   # Comma-separated policy IDs to run after tasks
  [--glm-plan]             # Enable GLM quota checking
  [--attempt]              # Attempt run (infer attempt number from feedback files)
  [--yes, -y]              # Skip confirmation prompt
```

### Policy Run Options

```
fa policy run <policy-id...>
  [--tool <tool>]          # AI tool (default: iflow)
  [--rounds <n>]           # Rounds (default: 3)

fa policy run-all
  [--tool <tool>]
  [--rounds <n>]
```

## Data Models

### Task (task.json)

```json
{
  "id": 5,
  "slug": "implement-auth",
  "parent_id": null,
  "status": "pending",
  "depends_on": [3, 4],
  "related_to": [7],
  "created_at": "2026-03-25T10:30:00",
  "completed_at": null
}
```

| Field          | Type           | Description                                                |
| -------------- | -------------- | ---------------------------------------------------------- |
| `id`           | int            | Global sequential task ID                                  |
| `slug`         | string         | URL-friendly identifier (alphanumeric + hyphens)           |
| `parent_id`    | int \| null    | Parent task ID for subtasks, null for standalone/parent    |
| `status`       | string         | `pending` \| `running` \| `completed`                      |
| `depends_on`   | int[]          | Task IDs for dependency graph (informational, no blocking) |
| `related_to`   | int[]          | Task IDs for related tasks (informational only)            |
| `created_at`   | string         | ISO 8601 timestamp                                         |
| `completed_at` | string \| null | ISO 8601 timestamp when completed                          |

### Task Directory Naming

Format: `{id}-{mm-dd}-{slug}`

- `id`: Global sequential ID
- `mm-dd`: Month-day of creation (zero-padded)
- `slug`: Task slug

Examples:

- `1-03-25-setup-project`
- `2-03-25-add-authentication`
- `15-04-01-refactor-api`

### Memory Files

After each task execution, the AI agent creates a `memory-N.md` file:

- `memory-1.md` - Created after first execution
- `memory-2.md` - Created after second execution (attempt mode)
- etc.

Memory files record the AI's work and decisions for future reference.

### Policy (YAML)

```yaml
id: testing
name: Test Compliance Check
description: Ensure test code complies with specifications
objective: |
  Ensure test code complies with specifications
specs:
  - "./specs/testing-guidelines.md"
  - "./specs/quality-guidelines.md"
scopes:
  required:
    - "git:./tests/"
    - "./src/"
  optional:
    - "./docs/"
  exclude:
    - "./tests/fixtures/"
    - "*.snap"
report:
  path: "./reports/policy/{{ policy.id }}/{{ date }}.md"
  template: |
    # Policy Report: {{ policy.name }}

    ## Summary
    ...

    ## Findings
    ...
```

| Field             | Type     | Description                                                  |
| ----------------- | -------- | ------------------------------------------------------------ |
| `id`              | string   | Unique policy identifier (filename stem if not specified)    |
| `name`            | string   | Human-readable name                                          |
| `description`     | string   | Brief description                                            |
| `objective`       | string   | Detailed objective for the AI agent                          |
| `specs`           | string[] | List of specification file paths (relative to project root)  |
| `scopes.required` | string[] | Required file scopes (`git:` prefix for git-aware filtering) |
| `scopes.optional` | string[] | Optional file scopes                                         |
| `scopes.exclude`  | string[] | Files/patterns to exclude from scopes                        |
| `report.path`     | string   | Report output path (supports Jinja2)                         |
| `report.template` | string   | Report template (Jinja2)                                     |

**Git-aware scoping**: Paths prefixed with `git:` are filtered to only include files that are:

- Staged in git
- Modified in working directory
- Untracked

## Execution Flow

### Task Execution (fa task run)

```
1. Find project root
2. Scan pending tasks
3. Filter by --start/--end if specified
4. Build execution plan (group tasks by parent)
5. Display task list and wait for user confirmation (skip with -y)
6. For each task in execution plan:
   - Standalone tasks: execute directly
   - Parent tasks with subtasks in range:
     i. Execute specified subtasks sequentially (sorted by ID)
     ii. Execute parent task (verification)
7. If --policies specified:
   a. For each policy ID:
      i. Load policy from .fa/policies/<id>.yml
      ii. Execute policy
```

### Parent Task Execution Rule

When executing subtasks (whether all or a subset via --start/--end):

- The parent task is **always executed after** all its subtasks complete
- Only the specified subtasks run (siblings outside the range are skipped)

Example: `fa task run --start 3 --end 3` where task 3 is a subtask of parent 2:

1. Execute subtask 3
2. Execute parent task 2 (verification)

### Single Task Execution

```
For each round (1 to --rounds):
1. Determine execution mode:
   - Default (fresh): attempt = 1
   - --attempt flag: attempt = count(feedback files) + 1
2. Build prompt with file paths
3. Invoke AI tool
4. Write agent output to .fa/logs/agents/{task-dir}/round-{n}-{tool}.log
5. AI creates memory-{attempt}.md in task directory
6. Check exit code
7. If failed, mark task as pending and continue to next task
```

### Attempt Number Inference

| Feedback Files Present                  | Attempt Number |
| --------------------------------------- | -------------- |
| None                                    | 1 (fresh run)  |
| `feedback-1.md`                         | 2              |
| `feedback-1.md`, `feedback-2.md`        | 3              |
| `feedback-1.md` through `feedback-N.md` | N + 1          |

### Prompt Building

The prompt contains **file paths**, not file contents. The AI agent reads the files itself.

#### Fresh Run (default, attempt = 1)

**Standalone task / Parent task (no subtasks)**:

```
## Task Information

- Task ID: 5
- Attempt: 1 (fresh run)
- Task file: .fa/tasks/5-03-25-implement-auth/task.md

Read the task description from the file above and complete it.
After completion, write your work summary to: .fa/tasks/5-03-25-implement-auth/memory-1.md
```

**Subtask**:

```
## Parent Task

- Task file: .fa/tasks/2-03-25-user-system/task.md

## Current Task

- Task ID: 3
- Attempt: 1 (fresh run)
- Task file: .fa/tasks/2-03-25-user-system/3-03-25-setup-db/task.md

Read the parent task for context, then complete the current task.
After completion, write your work summary to: .fa/tasks/2-03-25-user-system/3-03-25-setup-db/memory-1.md
```

#### Attempt Run (--attempt flag)

**Standalone task / Parent task (no subtasks)**:

```
## Task Information

- Task ID: 5
- Attempt: 3
- Task file: .fa/tasks/5-03-25-implement-auth/task.md

## Previous Attempts

### Memory Files

Read previous work summaries:
- .fa/tasks/5-03-25-implement-auth/memory-1.md
- .fa/tasks/5-03-25-implement-auth/memory-2.md

### Feedback

Read feedback from previous attempts:
- .fa/tasks/5-03-25-implement-auth/feedback-1.md
- .fa/tasks/5-03-25-implement-auth/feedback-2.md

Review the feedback and previous work, then continue the task.
After completion, write your work summary to: .fa/tasks/5-03-25-implement-auth/memory-3.md
```

**Subtask**:

```
## Parent Task

- Task file: .fa/tasks/2-03-25-user-system/task.md

## Current Task

- Task ID: 3
- Attempt: 3
- Task file: .fa/tasks/2-03-25-user-system/3-03-25-setup-db/task.md

## Previous Attempts

### Memory Files

Read previous work summaries:
- .fa/tasks/2-03-25-user-system/3-03-25-setup-db/memory-1.md
- .fa/tasks/2-03-25-user-system/3-03-25-setup-db/memory-2.md

### Feedback

Read feedback from previous attempts:
- .fa/tasks/2-03-25-user-system/3-03-25-setup-db/feedback-1.md
- .fa/tasks/2-03-25-user-system/3-03-25-setup-db/feedback-2.md

Review the feedback and previous work, then continue the task.
After completion, write your work summary to: .fa/tasks/2-03-25-user-system/3-03-25-setup-db/memory-3.md
```

### Policy Execution

Policies run after all tasks complete. Each policy:

1. Load policy YAML (with Jinja2 processing for includes)
2. Expand `git:` scoped paths
3. Apply exclude patterns
4. Build prompt from policy definition
5. Execute via AI tool for N rounds
6. Write report to specified path

## Tool Integration

### Supported AI Tools

| Tool     | Command Template                                                                              |
| -------- | --------------------------------------------------------------------------------------------- |
| iflow    | `iflow --prompt "<prompt>" -y --debug`                                                        |
| kilo     | `kilo run --auto "<prompt>" --print-logs --log-level DEBUG`                                   |
| claude   | `claude -p --dangerously-skip-permissions --output-format stream-json --verbose "<prompt>"`   |
| ccr      | `ccr code -p --dangerously-skip-permissions --output-format stream-json --verbose "<prompt>"` |
| opencode | `opencode run "<prompt>" --print-logs --log-level DEBUG`                                      |

### GLM Quota Management

When `--glm-plan` is enabled:

1. Check GLM token usage via API
2. If usage >= 70%, wait until reset time + 30 minutes
3. Proceed with execution

## Logging

### Log Structure

```
.fa/logs/
├── fa.log                           # Tool internal logs (DEBUG level)
└── agents/                          # Agent execution logs
    └── {task-dir}/
        ├── round-1-iflow.log
        ├── round-2-iflow.log
        └── ...
```

### Log Format

- Tool logs (fa.log): `{timestamp} [{level}] - {message}`
- Agent logs: Raw output from AI tool execution

### Log Levels

- Console: INFO level
- File (fa.log): DEBUG level

## Error Handling

1. **Task not found**: Print error, exit code 1
2. **Policy not found**: Print error, exit code 1
3. **Tool execution failure**: Log error, mark task pending, continue to next task
4. **Template not found**: Log error, skip task
5. **Git operations in non-git directory**: Treat all files as in-scope

## Implementation Notes

### Auto-initialization

Commands automatically create `.fa/` structure on-demand:

- `.fa/tasks/`
- `.fa/tasks/archive/`
- `.fa/logs/`
- `.fa/logs/agents/`
- `.fa/policies/`
- `.fa/templates/`

### Jinja2 Template Context

Available variables in `task_prompt.j2`:

| Variable             | Type           | Description                                  |
| -------------------- | -------------- | -------------------------------------------- |
| `task`               | dict           | Task metadata (from task.json)               |
| `task_file`          | string         | Path to task.md                              |
| `parent_file`        | string \| null | Path to parent's task.md (for subtasks)      |
| `attempt`            | int            | Current attempt number                       |
| `memory_files`       | list[string]   | List of memory file paths                    |
| `feedback_files`     | list[string]   | List of feedback file paths                  |
| `memory_output_path` | string         | Path for AI to write memory file             |
| `specs_dir`          | string         | Path to specs directory (PROJECT_ROOT/specs) |

### Subtask Discovery

A task has subtasks if its directory contains subdirectories matching pattern `{id}-{mm-dd}-{slug}` with valid `task.json`.

## Appendix: Command Examples

```bash
# Initialize project
fa init

# Create standalone task
fa task create implement-auth

# Create parent task
fa task create user-management-system

# Create subtask
fa task create setup-database --parent 2

# Create another subtask
fa task create implement-api --parent 2

# List all tasks
fa task list

# List pending tasks
fa task list --status pending

# Show task details
fa task info 5

# Run all pending tasks
fa task run

# Run all pending tasks without confirmation
fa task run -y

# Run specific task range with policies
fa task run --start 5 --end 10 --policies testing,security

# Run with attempt mode (after providing feedback)
fa task run --attempt

# Run with specific tool
fa task run --tool claude --rounds 5

# Mark tasks done
fa task done 5
fa task done 5-10

# Archive completed tasks
fa task archive 5-10

# Delete tasks
fa task rm 5
fa task rm 5-10 -f

# List policies
fa policy list

# Run single policy
fa policy run testing

# Run multiple policies
fa policy run testing security linting

# Run all policies
fa policy run-all
```
